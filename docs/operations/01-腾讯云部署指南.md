## 第一次部署（仅需做一次）

### 步骤 0：腾讯云控制台 → 安全组放行 80 端口

这一步**不在 Web Shell 里**做，而是在腾讯云**控制台网页**操作：

1. 登录腾讯云 → 进入你的服务器实例
2. 找到「**安全组**」→ 选择本机绑定的安全组 → 「**修改规则**」
3. 「入站规则」→「**添加规则**」：
   - **协议端口**：`TCP:80`
   - **源**：`0.0.0.0/0`（允许所有 IP）
   - **策略**：允许
4. 保存

> 这一步如果不做，手机访问会一直"连接被拒绝"。

### 步骤 1：登录服务器 Web Shell

腾讯云控制台 → 服务器实例 → 点「**登录**」按钮 → 选「**标准登录方式**」或类似 → 进入网页 Shell。

你应该看到一个黑底白字的终端界面。

### 步骤 2：安装系统基础软件

把下面**一整段**复制到 Web Shell 里执行（含 `&&` 是为了一条接一条不中断）：

```bash
sudo apt update && sudo apt upgrade -y && \
sudo apt install -y git nginx python3.11 python3.11-venv python3-pip curl ca-certificates ufw && \
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo bash - && \
sudo apt install -y nodejs
```

> 这一段耗时 1~3 分钟（首次更新系统比较慢）。结束后会看到命令行回到 `$` 提示符。

**验证**：

```bash
python3.11 --version    # 应该输出 Python 3.11.x
node --version          # 应该输出 v20.x.x
nginx -v                # 应该输出 nginx version: ...
```

### 步骤 3：拉取项目代码到 `/opt/gcp`

```bash
sudo mkdir -p /opt && sudo chown -R $(whoami) /opt && \
cd /opt && \
git clone https://github.com/Austin-Wang-1993/General_Communication_Platform.git gcp && \
cd gcp && \
ls -la
```

执行完应该能看到仓库根目录的文件列表。

#### 步骤 3.5：切到你要部署的分支（**首次部署必做**）

`git clone` 默认拉的是 `main` 分支。**首次部署一个新阶段（如 M0、M1...）时，相应代码可能还在 feature 分支上等你验收**——按双分支工作流（技术方案 §11.2）的约定。

下面把分支名替换成我告诉你的当前 feature 分支：

```bash
cd /opt/gcp && \
git fetch origin <分支名> && \
git checkout <分支名> && \
ls -la
```

例如 Agent 让你测 M1 时，分支名可能是 `cursor/m1-scenario-crud-a4d3`（**以 Agent 当次给出的为准**，勿照抄旧文档里的示例名）：

```bash
cd /opt/gcp && \
git fetch origin cursor/m1-scenario-crud-a4d3 && \
git checkout -B cursor/m1-scenario-crud-a4d3 origin/cursor/m1-scenario-crud-a4d3 && \
ls -la
```

执行后应看到 `backend/`、`frontend/`、`deploy/` 出现。

> **与 `pull-and-restart.sh` 的关系**：首次可仍按上面手动 `fetch + checkout`；之后每次更新请**优先**用下方「步骤 A」的一键脚本——脚本内已用 `git checkout -B` 对齐**本地分支名与远端**，并在重启后 **`curl` 本机健康检查**，避免「磁盘已是新代码、uvicorn 仍是旧进程」的假成功。

### 步骤 4：开放服务器自身防火墙（UFW）

```bash
sudo ufw allow OpenSSH && \
sudo ufw allow 'Nginx Full' && \
sudo ufw --force enable && \
sudo ufw status
```

> 应该看到 `Status: active` 且 80/443/22 端口已放行。

### 步骤 5：搭建后端 Python 环境

```bash
cd /opt/gcp/backend && \
python3.11 -m venv .venv && \
source .venv/bin/activate && \
pip install --upgrade pip && \
pip install -r requirements.txt && \
deactivate
```

> 耗时 30 秒~2 分钟。成功后 `/opt/gcp/backend/.venv/` 目录已建好。

---

### 步骤 6：构建前端静态产物

```bash
cd /opt/gcp/frontend && \
npm ci && \
npm run build && \
ls -la dist/
```

> 耗时 1~3 分钟。成功后 `dist/index.html` 与若干 JS/CSS 文件出现。

---

### 步骤 7：写入环境变量配置（含 DeepSeek Key）

**重要：把下面的 `<把你的真实 key 贴在这一行>` 替换成你的真实 DeepSeek API Key**（形如 `sk-xxxxx`）。

```bash
sudo mkdir -p /etc/gcp && \
sudo tee /etc/gcp/backend.env > /dev/null <<'EOF'
DEEPSEEK_API_KEY=<把你的真实 key 贴在这一行>
DEEPSEEK_MODEL=deepseek-chat
GCP_DATA_DIR=/opt/gcp/backend/data
CORS_ORIGINS=http://43.155.205.89
DIALOGUE_HISTORY_LIMIT=20
GCP_LOG_LEVEL=INFO
EOF
sudo chmod 600 /etc/gcp/backend.env && \
sudo cat /etc/gcp/backend.env
```

> ⚠️ 复制 `EOF` 那段时一定要包括 `EOF`（结束标记），否则命令不会闭合。  
> ⚠️ Key 写错的话之后所有 LLM 调用都会失败。可以稍后用 `sudo nano /etc/gcp/backend.env` 修改。

---

### 步骤 8：注册后端为 systemd 服务并启动

```bash
sudo cp /opt/gcp/deploy/gcp-backend.service /etc/systemd/system/ && \
sudo systemctl daemon-reload && \
sudo systemctl enable --now gcp-backend && \
sudo systemctl status gcp-backend --no-pager
```

应该看到状态 `active (running)`。如果不是 active：

```bash
sudo journalctl -u gcp-backend -n 50
```

会打印最近 50 条日志，可以发给我排查。

**自检（与 `pull-and-restart.sh` 检测一致）**：在普通用户下执行：

```bash
systemctl show gcp-backend.service -p LoadState --value
```

应输出 **`loaded`**。若是 **`not-found`**，多半是只拷贝了文件但尚未 `daemon-reload`，请再执行：

```bash
sudo systemctl daemon-reload && sudo systemctl enable --now gcp-backend
```

> 说明：旧版同步脚本用 `list-unit-files | grep` 判断服务是否存在，在部分环境下会误判；现已改为用 `LoadState` + 自动 `daemon-reload`。你拉取最新 `main` 后重跑 `./deploy/pull-and-restart.sh` 即可。

### 步骤 9：配置 Nginx 反代并 reload

```bash
sudo cp /opt/gcp/deploy/nginx-gcp.conf /etc/nginx/sites-available/gcp && \
sudo ln -sf /etc/nginx/sites-available/gcp /etc/nginx/sites-enabled/gcp && \
sudo rm -f /etc/nginx/sites-enabled/default && \
sudo nginx -t && \
sudo systemctl reload nginx
```

> `nginx -t` 应该输出 `syntax is ok` 和 `test is successful`。

## 一键同步（推荐 · 复制一整段即可）

> 适合：**不想记多步命令**、希望**一次粘贴**就完成「拉最新部署脚本 + 切到要测的分支 + 全量依赖 + 构建 + 重启 + 自检」。

仓库提供入口脚本 **`deploy/server-one-shot-sync.sh`**：会先 `git reset --hard origin/main`（保证 `deploy/` 里的脚本自身为 GitHub 最新），再调用 `pull-and-restart.sh` 部署你指定的远端分支；并默认 **`GCP_SYNC_FULL=1`**（每次 `pip install -r` + `npm ci`，更慢但更稳）。

### ① 测 Agent 给你的 feature 分支（合并进 main 之前）

把下面**整段**复制到服务器 Web Shell，**只改最后一行里的分支名**（与 GitHub 上完全一致）：

```bash
cd /opt/gcp && bash deploy/server-one-shot-sync.sh 'cursor/这里改成-Agent-给你的分支名'
```

### ② 测试通过、Agent 已合并进 main 之后 —— 同步正式基线

把下面**整段**复制即可（**不用改**）：

```bash
cd /opt/gcp && bash deploy/server-one-shot-sync.sh main
```

### 说明

- 前提：服务器上已完成本文 **步骤 1～9**（尤其 **步骤 8 Nginx、步骤 8 systemd 后端**），且仓库在 **`/opt/gcp`**。若仓库路径不同，可在命令前加：`export GCP_REPO_DIR=/你的路径`。
- 若想加快（跳过每次全量 pip/npm），可：`export GCP_SYNC_FULL=0` 后再执行（一般仍建议默认全量）。
- 若脚本红字 `✗` 退出，表示**没有部署成功**，把终端截图发给 Agent 即可。

## 之后每次更新的流程

### 重要：为什么「调试页变了」不等于「后端已更新」？

1. **Nginx** 对 `http://43.155.205.89/debug/` 使用 `alias`，**直接读磁盘**上的 `backend/app/debug_ui/` 静态文件；`git` 更新后，即使**不重启** `gcp-backend`，刷新浏览器也常能立刻看到新的调试页 HTML/JS。
2. **`/api/`** 会反代到本机 **`127.0.0.1:8000`** 的 **uvicorn（gcp-backend）** 进程；Python 路由驻留在**内存**里，**必须 `systemctl restart gcp-backend`** 才会加载新代码。
3. 若某次只更新了磁盘、或 `gcp-backend` 未注册导致脚本**跳过了重启**，就会出现「界面像新的、`POST /api/v1/...` 仍 `404`」。新版 `pull-and-restart.sh` 把**健康检查失败视为整次部署失败**（非 0 退出），并在终端打印**当前本地分支名**便于你与 Agent 核对。

我（Cloud Agent）在 feature 分支推送新代码后，你在 Web Shell 里：

### A. 拉 feature 分支测试

```bash
cd /opt/gcp && bash deploy/server-one-shot-sync.sh 'cursor/<Agent-给你的分支名>'
```

> 与上文「一键同步」① 相同。若你已手动把 `deploy/` 更新到最新，也可简写：`cd /opt/gcp && ./deploy/pull-and-restart.sh cursor/<分支名>`（默认不全量 pip/npm，除非设置 `GCP_SYNC_FULL=1`）。

### B. 测试通过后我合并到 main，你拉 main 作基线

```bash
cd /opt/gcp && bash deploy/server-one-shot-sync.sh main
```

> 与上文「一键同步」② 相同。或不带一键入口：`cd /opt/gcp && ./deploy/pull-and-restart.sh`（默认拉 main）。

### C. 测试不通过怎么办

告诉我具体哪里不对——截图、文案、控制台报错都行。我修了再推一个新 commit / 新分支。你重跑步骤 A。

如果某次 feature 把服务搞挂了，立刻回滚到 main：

```bash
cd /opt/gcp && ./deploy/pull-and-restart.sh
```

## 常见操作速查

### 查看后端日志

```bash
sudo journalctl -u gcp-backend -f      # 实时
sudo journalctl -u gcp-backend -n 100  # 最近 100 行
```

### 查看 Nginx 日志

```bash
sudo tail -f /var/log/nginx/access.log
sudo tail -f /var/log/nginx/error.log
```

### 修改 DeepSeek Key

```bash
sudo nano /etc/gcp/backend.env
# 修改 DEEPSEEK_API_KEY 那行，Ctrl+O 保存，Ctrl+X 退出
sudo systemctl restart gcp-backend
```

### 看场景包数据

```bash
ls -la /opt/gcp/backend/data/scenarios/         # 全部场景包
cat /opt/gcp/backend/data/scenarios/index.json  # 摘要索引
```

### 整个重启（终极方案）

```bash
sudo systemctl restart gcp-backend && sudo systemctl reload nginx
```

### 服务器磁盘占用查询

```bash
df -h                                # 总体磁盘
du -sh /opt/gcp/backend/data/        # 场景包数据目录占用
```

## 排错指南

| 症状 | 可能原因 | 检查 / 修复 |
|---|---|---|
| 手机访问"连不上服务器" | 安全组没放行 80 端口 | 控制台 → 安全组 → 入站加 TCP:80 |
| `nginx -t` 报错 | 配置文件语法错 | 看具体报错；通常是新加配置改坏 |
| `gcp-backend` 起不来 | Python 依赖没装、端口被占用 | `journalctl -u gcp-backend -n 50` |
| 前端页面 404 | dist 没构建出来 | 重跑 `cd /opt/gcp/frontend && npm run build` |
| API 调用返回 502 | 后端没运行或端口不通 | `systemctl status gcp-backend` + 看日志 |
| 健康检查 `deepseek_configured: ✗` | 步骤 7 Key 没正确写入 | 重新 `cat /etc/gcp/backend.env`，再 `systemctl restart gcp-backend` |
| 调试页/前端已更新，但新 API 仍 `404` / `Not Found` | **gcp-backend 进程仍是旧代码**（未重启成功），或 Nginx 反代未指到当前 uvicorn；调试页走磁盘故可先变新 | 在服务器执行：`systemctl status gcp-backend`；`curl -sS http://127.0.0.1:8000/api/v1/health`；`sudo journalctl -u gcp-backend -n 80`。确认 `./deploy/pull-and-restart.sh <分支>` **整段无报错结束**（脚本现已强制本机健康检查） |
