# 代码架构与目录约定

> **文档版本**：v0.1.5  
> **更新时间**：2026-05-14  
> **关联**：`../product/01-产品需求文档.md`（PRD v0.5.4）、`../product/02-前端需求文档.md`（v0.2.5）、`01-技术方案.md`（v0.3.6）、`03-API 接口文档.md`（v0.1.8）、`04-业务流程与状态机.md`  
> **本文要回答的问题**：业务逻辑、数据存取、前端交互、外部 API 调用——这些代码到底**放在哪里**？互相**怎么调**？前端字段与后端字段**怎么对齐不出错**？  
> **本文给非技术读者的核心信息**：项目就像一栋有 4 层的房子，每层只做自己的事；什么代码住哪一层，本文白纸黑字写清楚。开工之后任何"这段代码该写在哪儿？"的问题，先翻这份文档。

## 0. 设计原则（一句话三条）

1. **分层（Layering）**：把代码切成几层，每层只关心一件事；上层调下层，下层不调上层。
2. **封装（Encapsulation）**：相关代码放在同一个文件 / 类 / 模块里，外部通过明确的函数签名调用，不直接戳进它的内部。
3. **前后端契约（Contract）**：前后端约定好字段名、类型、错误格式，**任何一方擅自改动都视为破坏契约**，必须双方同步。

## 1. 总体视图

```text
                       ┌────────────────────────────────────┐
                       │       手机浏览器 (React SPA)        │
                       │                                    │
   PAGE  ─────────►    │  ① 页面层 Pages   (P1, P2, ..., P3)│
                       │            ↓                        │
   COMPONENT ───────►  │  ② 组件层 Components               │
                       │            ↓                        │
   HOOK  ─────────►    │  ③ 业务钩子层 Hooks  (use*)         │
                       │            ↓                        │
   SERVICE  ──────►    │  ④ 服务客户端 Services (fetch API)  │
                       └────────────────┬───────────────────┘
                                        │   HTTP JSON (首版 IP 直连无 HTTPS;
                                        │              详见 01-技术方案.md §11)
                                        ▼
                       ┌────────────────────────────────────┐
                       │  Nginx (反代 /api 到 FastAPI)        │
                       └────────────────┬───────────────────┘
                                        ▼
                       ┌────────────────────────────────────┐
                       │       后端 (FastAPI Python)         │
   ROUTER ────────►    │  ① 路由层 Router                    │
                       │            ↓                        │
   SERVICE ───────►    │  ② 业务服务层 Service               │
                       │            ↓             ↘          │
   REPOSITORY ────►    │  ③ 数据仓库层 Repo       LLM 调用    │
                       │            ↓             ↓           │
   CLIENT ────────►    │     文件系统 data/   ④ LlmClient    │
                       │     (JSON 文件)         (DeepSeek)   │
                       └────────────────────────────────────┘
```

**总规则**：箭头**只能向下**。下层**绝对不能**调用上层，否则就乱套（俗称"循环依赖"）。

## 2. 后端分层（FastAPI Python）

### 2.1 ① 路由层 Router

**职责**：

- 接收 HTTP 请求。
- 用 Pydantic 模型校验请求参数（长度、类型、枚举）。
- 调一个业务服务的方法，**只调一个**——这一层不写业务规则。
- 把业务层返回的 Pydantic 对象包装成 HTTP JSON 响应。
- 把业务层抛出的异常翻译成对应 HTTP 状态码（4xx / 5xx）。

**禁止**：

- ❌ 在 Router 里直接读写 JSON 文件。
- ❌ 在 Router 里直接调 DeepSeek。
- ❌ 在 Router 里实现"如果用户没填 X 就要怎样"这种业务判断（应该交给 Service）。

**目录**：`backend/app/routers/`

```text
backend/app/routers/
├── scenario_packages.py   PRD §5.5 / §6.1 包 CRUD + commit-intake
├── creation_jobs.py       PRD §6.2~§6.5 异步 Job + 进度查询
├── runtime.py             PRD §6.6 + §6.6.5 enter / turns / auto-opener
├── hints.py               PRD §6.7
├── analytics.py           PRD §6.8
└── health.py              健康检查
```

### 2.2 ② 业务服务层 Service

**职责**：

- 实现 PRD §6.1~§6.8 全部业务规则。
- 接收**已校验**的 Pydantic 对象，返回 Pydantic 对象。
- 调用 Repository 读写数据；调用 LlmClient 调 DeepSeek；调用 Validators 跑业务校验。
- 抛出**业务异常**（如 `LifecyclePhaseError`、`InvalidTurnError`），交给 Router 翻译成 HTTP 错误。

**禁止**：

- ❌ 不知道 HTTP、不引用 FastAPI 任何东西（这样 Service 单独可测）。
- ❌ 不直接打开文件、不直接发 HTTP 请求（这是 Repo 和 Client 的事）。

**目录**：`backend/app/services/`

```text
backend/app/services/
├── intake_service.py          §6.1 五字段扩写
├── framework_service.py       §6.2 章节骨架
├── roster_service.py          §6.3 角色生成
├── narrative_service.py       §6.4 单节剧情扩写
├── mission_service.py         §6.5 单节任务
├── dialogue_service.py        §6.6 用户发 → NPC 续聊，含 §6.6.4 全部规则
├── auto_opener_service.py     §6.6.5 进节自动开场
├── hint_service.py            §6.7 R1
├── analytics_service.py       §6.8 R2
├── job_runner.py              §6.2~§6.5 异步 Job 编排，调上面 5 个 service
└── pointer_service.py         §5.5/§5.6 指针更新（enter / 切节）
```

每个 service 是一个 `class`，构造函数注入 Repo 和 Client（依赖注入），便于测试时替换为假实现。

### 2.3 ③ 数据仓库层 Repository

**职责**：

- 把"业务对象"（Pydantic 模型）和"`data/` 下的 JSON 文件"互相转换。
- 提供原子读写、按包加 asyncio 锁。
- 暴露**只关心数据存取**的方法签名，如：

  ```python
  await package_repo.load(scenario_id) -> ScenarioPackage
  await package_repo.save(package) -> None
  await turn_repo.append(scenario_id, ch, sec, turn) -> None
  await turn_repo.list_section(scenario_id, ch, sec, limit=20) -> list[Turn]
  ```

**禁止**：

- ❌ 不知道业务规则（不知道"NPC 不能自己跟自己说话"这种事）。
- ❌ 不调 LLM。
- ❌ 不引用 FastAPI。

**好处**：未来如果想从"JSON 文件"换成"SQLite"或"PostgreSQL"，**只动这一层**，其他层完全不变。

**目录**：`backend/app/repositories/`

```text
backend/app/repositories/
├── base.py              通用 JSON 读写工具、按包加锁
├── package_repo.py      package.json
├── intake_repo.py       intake.json
├── analysis_repo.py     analysis.json
├── framework_repo.py    framework.json
├── roster_repo.py       roster.json
├── narrative_repo.py    sections/ch*_sec*/narrative.json
├── mission_repo.py      sections/ch*_sec*/mission.json
├── turn_repo.py         sections/ch*_sec*/turns.jsonl
├── hint_repo.py         sections/ch*_sec*/hints/{turn_id}.json
├── analytics_repo.py    sections/ch*_sec*/analytics.json
└── job_repo.py          jobs/{job_id}.json
```

### 2.4 ④ 外部客户端层 Client

**职责**：

- 和外部世界打交道。**目前只有 DeepSeek**。

- 实现**超时**、**重试**、**错误分类**（网络错误 vs API 错误 vs 输出 schema 错误）。

**禁止**：

- ❌ 不知道业务规则——它只知道"把消息发给 DeepSeek，把响应字符串拿回来"。

**目录**：

## 4. 前后端契约（最容易出问题的地方）

### 4.1 字段一一对应（命名规范）

- **后端 Pydantic 字段名** = **JSON 键名** = **前端 TypeScript 字段名**。
- 全部用 `snake_case`（如 `speaker_id`、`expects_user_response`、`current_chapter_id`）。**禁止**前端单独转 `camelCase`——多一层映射多一层翻车。
- 例：

  ```ts
  interface Turn {
    scenario_id: string;
    chapter_id: number;
    section_id: number;
    turn_id: string;
    created_at: string;
    speaker_id: string;
    recipient_id: string;
    content: string;
    expects_user_response: boolean;
    turn_writer: 'model_npc' | 'human_user';
  }
  ```

  对应后端：

  ```python
  class Turn(BaseModel):
      scenario_id: str
      chapter_id: int
      section_id: int
      turn_id: str
      created_at: str
      speaker_id: str
      recipient_id: str
      content: str
      expects_user_response: bool
      turn_writer: TurnWriter
  ```

### 4.2 错误响应统一格式

后端任何 4xx / 5xx 响应都返回：

```json
{
  "error_code": "lifecycle_phase_invalid",
  "message": "场景包尚未完成创作，无法进入对话页",
  "details": { "current_phase": "creation_running" }
}
```

前端 `apiClient.ts` 捕获后抛 `ApiError`，业务 Hooks 统一处理（弹 Toast + 决定是否重试）。

### 4.3 类型同步策略（首版）

- **首版手动维护**：每改一次后端 Pydantic，**立即**同步改对应的前端 `types/`。两边字段一字不差。
- **不引入自动代码生成**（如 OpenAPI Generator）：单人开发场景下，工具成本 > 收益。等智能体/字段成形且稳定后，再考虑用 FastAPI 自动生成 OpenAPI YAML → 前端用 `openapi-typescript` 反向生成 types。
- **PR Review 时**：任何后端 model 改动，**Reviewer（你）必须看到对应的前端 types 改动**；否则视为"破坏契约"。

### 4.4 路由前缀

后端所有业务接口前缀 `/api/v1/`。前端 `apiClient.ts` 中拼接 `import.meta.env.VITE_API_BASE` + `/api/v1`。首版 `VITE_API_BASE` 默认空字符串（与前端同域），生产部署时 Nginx 反代到后端。

## 5. 测试策略（首版极简）

| 层 | 测试类型 | 工具 | 覆盖目标 |
|---|---|---|---|
| 后端 Validators | 单元测试 | `pytest` | PRD §6.6.4 等硬规则的每一条 |
| 后端 Service | 单元测试 + 假 Repo/Client | `pytest` | 业务流程，不打真 LLM |
| 后端 Repository | 单元测试 | `pytest` + 临时目录 | JSON 读写 round-trip |
| 后端 Router | 集成测试 | `httpx.AsyncClient` + `pytest-asyncio` | 端到端 happy path |
| 前端 Service | 单元测试（可选） | `vitest` + MSW | API 客户端的请求体/错误处理 |
| 前端 Hook/Component | 跳过首版 | — | 等业务稳定再补；调试页本身就是"手动 E2E" |

**首版目标**：Validators + Service + Repository 各覆盖**主路径**，跑一遍 < 30 秒。Router 集成测试覆盖每个智能体一个 happy path。**不**追求覆盖率指标。

## 6. 命名约定（开工前定好，免得返工）

| 项 | 约定 | 示例 |
|---|---|---|
| 文件名 | `snake_case.py` / `kebab-case.tsx` | `dialogue_service.py` / `ChatMessage.tsx` |
| Python 类 | `PascalCase` | `DialogueService`, `Turn` |
| Python 函数/变量 | `snake_case` | `continue_npc`, `prior_turns` |
| TS interface / type | `PascalCase` | `Turn`, `ScenarioPackage` |
| TS 函数/变量 | `camelCase` | `sendMessage`, `currentSection` |
| API 路径 | `kebab-case` 复数资源 | `/scenario-packages/{id}/sections/{ch}/{sec}/turns` |
| JSON 键 | `snake_case` | `speaker_id`, `current_chapter_id` |
| 分支名 | `cursor/<topic>-a4d3` | `cursor/m0-skeleton-a4d3` |
| Commit | `<type>(<scope>): <subject>` | `feat(dialogue): NPC 续聊支持 G8` |

`<type>` ∈ `feat / fix / docs / refactor / test / chore / perf / style`。

## 7. 这套架构如何应对未来变化

| 未来变化 | 影响层 |
|---|---|
| 改 P3 聊天气泡样式（圆角变成方角） | 仅 `components/chat/ChatMessage.tsx` |
| 加一个智能体（如 §6.9 全旅程报告） | 后端：新加 `services/journey_report_service.py` + `repositories/...` + `routers/...`；前端：新加 hook + service + 弹窗组件 |
| 持久化从 JSON 文件换成 PostgreSQL | 仅 `backend/app/repositories/` 内部实现重写，Service 完全不动 |
| 换 LLM 模型 / 换厂商 | 仅 `backend/app/clients/llm_client.py` 与 `prompts/*.md` |
| 字段名调整（如 `speaker_id` 改成 `speaker_character_id`） | 后端 model + 前端 types **同步**改；服务层与 UI 跟着改；**必须当作"破坏契约"在 PR Review 时确认** |
| 加用户账号系统 | 后端：在 Router 加鉴权中间件 + 数据按 user_id 分区；其他层基本不动 |

## 8. 给非技术读者的"为什么这样分层"小结

- 当你在产品体验上发现"这条消息样式不对"——我只动**组件层**一个文件，5 分钟改完，不可能影响业务规则。
- 当你说"NPC 续聊规则应该改"——我只动**业务服务层**一个文件，PRD 上的硬规则与代码一一对应，看代码就像看 PRD。
- 当你说"数据丢了想从备份恢复"——所有数据都在 `data/` 目录的 JSON 文件，直接 cp 进去就行；**业务代码与数据格式解耦**。
- 当 DeepSeek 涨价想换厂商——我只动**外部客户端层**一个文件，提示词模板不变。
- **每一层都可以独立测试**——不用启动整个系统就能验证一条业务规则；这对"我这边没法本地开发"的工况尤其重要，bug 容易在 Cloud Agent 提交前就被单元测试拦住。
