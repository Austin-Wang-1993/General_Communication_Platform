## 2. 目标与范围（首版可交付）

**目标**：跑通「单用户、多场景包隔离」下的 **§6.1～§6.6 主链** + **§6.6.5 进节自动开场** + **§6.7 / §6.8 双轨评估**（按 PRD §8 分期），前端满足前端文档 P1～P3a 与 DBG 入口策略。

**首版建议范围**

- 单租户、单后端进程；创作期与运行期 API 明确分离。  
- **§5.7**：每小节 `section_narrative` / `section_mission` **独立存储**；**禁止**单次 LLM 调用落盘全书小节。  
- **§5.4 生命周期 6 状态**：`draft / intake_committed / creation_running / creation_failed / creation_succeeded / runtime_active`。**不**实现 `runtime_complete` 与 `archived`；**不**实现"旅程结束"接口与 UI。  
- **§5.6 章节自由推进**：运行期允许 P3a 任意切节（指针直接改 `chapter_id`/`section_id`，不清空任何小节历史）；数据读写仍只针对**当前指针**小节。  
- **§6.6.5 进节自动开场**：实现"用户进入 `turn === 0` 的小节时由后端自动生成首条 NPC 回合"的接口与触发逻辑。

**非目标（首版可明确不做）**

- 多租户账号、计费、消息队列集群、全旅程报告（PRD §4 已说明可另文）。  
- 语音实时流、离线客户端。  
- "清空本节对话记录"功能（PRD §5.6 已允许但首版不实现）。  
- 场景包"修改基本信息"或"重命名"功能（首版只能新建/删除）。

## 3. 技术选型（确定）

| 层级 | 选型 | 说明 |
|------|------|------|
| 前端 | **Vite + React + TypeScript** | SPA；`services/` 封装 API；路由对应 P1～P3a；**移动端优先**（前端文档 G12） |
| UI 组件 | **自写组件 + Tailwind CSS** | 轻量、移动端友好；不引入庞大 UI 库；颜色/字体由本仓自定义设计 token |
| 状态管理 | **TanStack Query (React Query) + Zustand** | TanStack Query 管理服务端数据与轮询；Zustand 管理本地 UI 状态（当前指针、草稿等） |
| 后端 | **Python 3.11+** | 编排、校验、文件 I/O |
| Web 框架 | **FastAPI + Uvicorn** | REST；任务进度暂时**用轮询而非 SSE**（实现简单，单用户够用） |
| LLM | **DeepSeek（`deepseek-chat` 模型）** | 首版**全部智能体**统一使用 `deepseek-chat`（DeepSeek-V3 系列，通用对话模型）；Key 仅服务端环境变量 `DEEPSEEK_API_KEY`；HTTP 客户端 `httpx`；薄封装、无 LangChain。若后续个别智能体（如剧情扩写、本节复盘）质量不达预期，再单独切到 `deepseek-reasoner` 推理模型 |
| 提示词 | **按智能体分 Markdown** | `backend/app/prompts/` 下每个 agent 一个 `.md`，与 PRD §6.1~§6.6.5 一一对应 |
| 持久化（首版） | **纯文件 + JSON** | 单用户够用、可视化好（PM 可直接打开 JSON 查看 LLM 生成内容）；目录结构 `data/scenarios/{scenario_id}/...` 每个小节一份独立 JSON。**不使用 SQLite**，避免引入额外查询/迁移成本 |
| 部署 | **腾讯云 Ubuntu + Nginx + systemd** | 静态 `dist` + 反代 `/api`；首版**无域名、纯 IP + HTTP 直连**（用户实际目标：`http://43.155.205.89/`），不申请 HTTPS 证书（Let's Encrypt 不签发给纯 IP；自签证书在手机端会一直弹"不安全"，反而劝退）；后续若买域名再启用 HTTPS |
| 仓库与 CI | **GitHub 远程 + 服务器手动 pull** 或简单 webhook | 详见 §11 部署链路 |

## 5. 核心数据模型（文件 + JSON 落地）

**首版采用纯文件 + JSON 持久化**，目录结构如下（`data/` 为后端工作目录下的数据根）：

```text
data/
└── scenarios/
    ├── index.json                         全部 scenario_id 列表 + 简要摘要（P2 列表用，避免遍历目录）
    └── {scenario_id}/
        ├── package.json                   §5.5 包级元数据
        ├── intake.json                    §6.1 输入快照（五字段，commit 后不可变）
        ├── analysis.json                  §6.1 扩写输出
        ├── framework.json                 §6.2.3 章节骨架
        ├── roster.json                    §6.3.2 角色表
        ├── sections/
        │   └── ch{chapter_id}_sec{section_id}/
        │       ├── narrative.json         §6.4.2 单小节叙事
        │       ├── mission.json           §6.5.2 单小节任务
        │       ├── turns.jsonl            §6.6.3 该小节回合流水（JSONL，每行一条 turn）
        │       ├── hints/
        │       │   └── {linked_turn_id}.json   §6.7 单条提示对象
        │       └── analytics.json         §6.8 最新一次成功复盘（覆盖写）
        └── jobs/
            └── {job_id}.json              §7 长任务进度（framework / world）
```

### 5.1 `package.json`（包根）

| 字段 | 类型 | 说明 |
|------|------|------|
| `scenario_id` | UUID 字符串 | PRD §5.2 |
| `scenario_title` | string | §5.5 / §6.1 |
| `user_display_name` | string | §6.1 |
| `lifecycle_phase` | enum **6 值** | **§5.4**（仅 6 值，**不**含 `runtime_complete`/`archived`） |
| `current_chapter_id` / `current_section_id` | int 或 null | §5.5；运行期非 null |
| `runtime_awaiting_user` | bool 或 null | 运行期是否等待用户回复 |
| `created_at` / `updated_at` | RFC3339 | §5.3 |

### 5.2 创作产物（分文件，按小节分目录）

| 实体 | 粒度 | 文件路径 |
|------|------|----------|
| `intake_snapshot` | 每包一份 | `intake.json` |
| `scenario_analysis` | 每包一份 | `analysis.json` |
| `story_framework` | 每包一份 | `framework.json` |
| `character_roster` | 每包一份 | `roster.json` |
| `section_narrative` | **每小节一份** | `sections/ch{c}_sec{s}/narrative.json` |
| `section_mission` | **每小节一份** | `sections/ch{c}_sec{s}/mission.json` |
| `turn` | **每小节追加** | `sections/ch{c}_sec{s}/turns.jsonl`（按 `created_at` 追加；首条由 §6.6.5 自动开场写入） |
| `user_reply_hint` | 每条一份 | `sections/ch{c}_sec{s}/hints/{linked_turn_id}.json` |
| `section_analytics` | 每小节最新一份 | `sections/ch{c}_sec{s}/analytics.json`（成功时覆盖；失败时不写） |

**并发策略**：首版**单用户**，对每个 `scenario_id` 使用**进程内 asyncio.Lock**（按包 ID 加锁）即可；不引入数据库行锁或文件锁。多用户阶段再升级到数据库。

## 6. API 雏形（REST 分组）

前缀统一 **`/api/v1`**。下列为**资源向**设计，路径可微调，**字段名**建议与 PRD JSON 键一致，减少映射层。

### 6.1 场景包与入口

| 方法 | 路径 | 作用 |
|------|------|------|
| `POST` | `/scenario-packages` | 创建包，返回 `scenario_id`，初始 `lifecycle_phase=draft` |
| `GET` | `/scenario-packages` | P2 列表 |
| `GET` | `/scenario-packages/{scenario_id}` | 包根 + 可选摘要 |
| `DELETE` | `/scenario-packages/{scenario_id}` | P2b 删除 |
| `POST` | `/scenario-packages/{scenario_id}/commit-intake` | 五字段校验并锁定 → `intake_committed` |

### 6.2 创作流水线（与 P2.2 / P2.4 / G3 / G4 对齐）

**方案 A（推荐 Demo）**：**异步 Job + 轮询 / SSE**

| 方法 | 路径 | 作用 |
|------|------|------|
| `POST` | `/scenario-packages/{id}/jobs/framework` | 启动 §6.2（及编排上的 §6.3）；返回 `job_id` |
| `GET` | `/scenario-packages/{id}/jobs/{job_id}` | 状态 + **`current_step_label`**（G10）+ `progress_hint`（可选） |
| `POST` | `/scenario-packages/{id}/jobs/world` | 启动全书小节循环 §6.4+§6.5（每节独立调用 LLM） |
| `POST` | `/scenario-packages/{id}/jobs/{job_id}/cancel` | 通用取消接口（对齐 P2.2、P2.4 取消语义；具体清库范围按 job.type 由后端决定） |

> **本表只列方法+路径概要**；完整的请求体 / 响应体 / 错误码契约见 **`03-API 接口文档.md`**——以该文档为权威。本表与 API 文档不一致时，**以 API 文档为准**。

**方案 B（简化）**：同步 HTTP 仅适合极短链路；**不推荐**用于全书小节生成。

**`POST .../commit-intake` 请求体（示例）**

```json
{
  "scenario_title": "string",
  "user_display_name": "string",
  "scene_brief": "string",
  "user_goal_brief": "string",
  "vocabulary_list": "string",
  "force_reset_creation": true
}
```

- `force_reset_creation`：当包内已有 `story_framework` 且用户从 P2.1 再次提交时，**G3** 置 true，后端原子删除下游产物后再跑新 job。

### 6.3 运行期（P3）

| 方法 | 路径 | 作用 |
|------|------|------|
| `GET` | `/scenario-packages/{id}/runtime` | `lifecycle_phase`、`current_*`、`runtime_awaiting_user`、当前节 `turns` 数量摘要等 |
| `POST` | `/scenario-packages/{id}/sections/{ch}/{sec}/enter` | 进入运行态：将指针更新为 `(ch, sec)`，若该节 `turns.jsonl` 不存在或为 0 行，**同步触发 §6.6.5 自动开场**（在响应中直接返回首条 NPC 回合）；若该节已有回合，仅切指针，不重复触发 |
| `POST` | `/scenario-packages/{id}/sections/{ch}/{sec}/auto-opener` | **显式触发**进节自动开场（用于前端"重试本节开场"按钮）；仅当该节 turn 数 === 0 时执行，否则返回 409 |
| `GET` | `/scenario-packages/{id}/sections/{ch}/{sec}/turns` | 本节全量聊天记录（升序） |
| `POST` | `/scenario-packages/{id}/sections/{ch}/{sec}/turns` | body 为单条用户发言：`content`、`recipient_id`；服务端写 `turn_writer=human_user` 并**同步触发** NPC 生成链返回新增的 NPC 回合 |
| `POST` | `/scenario-packages/{id}/sections/{ch}/{sec}/hints` | body：`target_turn_id`；仅当等待用户时 200，否则 409 |
| `GET` | `/scenario-packages/{id}/sections/{ch}/{sec}/hints/latest` | 读当前绑定建议 |
| `POST` | `/scenario-packages/{id}/sections/{ch}/{sec}/analytics` | 触发 §6.8；同步返回 ready/failed |

> **说明**：原 `sections/{ch}/{sec}/pointer` 单独切节接口合并入 `enter`——切节本身就是"进入该节"，对前端 P3a 而言一次请求完成"切指针 + 必要时触发自动开场 + 拉历史"。

**`POST .../enter` 响应体（示例）**

```json
{
  "current_chapter_id": 1,
  "current_section_id": 1,
  "runtime_awaiting_user": true,
  "section_narrative": { ... },
  "section_mission": { ... },
  "turns": [ /* 该节全部历史，含本次自动开场新写入的首条（若有） */ ],
  "auto_opener_triggered": true
}
```

**`POST .../turns` 请求体（示例）**

```json
{
  "content": "English text...",
  "recipient_id": "npc_pm_lead"
}
```

**`POST .../turns` 响应体**：返回服务端新写入的**用户回合**与**后续生成的 NPC 回合数组**（可能 1~3 条，受 §6.6.4 NPC–NPC 上限约束）；前端按序追加渲染。

**单条 `turn` 响应字段**：须包含 PRD §6.6.3 全部键（含 `turn_writer`）。

### 6.4 错误与枚举

- HTTP **409**：业务状态不允许（如非等待用户却发 hint）。  
- HTTP **422**：校验失败（长度、枚举外值、`speaker_id`/`recipient_id` 非法等）。  
- 响应体统一 `{"error_code","message","details?}`，便于前端 Toast。

## 7. 长任务与 G10 进度

- **Job 落地**：`data/scenarios/{scenario_id}/jobs/{job_id}.json`，字段：`job_id`、`scenario_id`、`type`（`framework` | `world`）、`status`（`queued` | `running` | `succeeded` | `failed` | `canceled`）、`current_step_label`、`progress_hint`（可选，如"第 2/5 节"）、`error_message?`、`created_at`、`updated_at`。  
- **后台任务**：FastAPI 的 `BackgroundTasks` + `asyncio.create_task`；不引入 Celery / Redis（首版单进程够用）。  
- **轮询而非 SSE**：前端 P2.2 / P2.4 每 **1s** 轮询 `GET .../jobs/{id}`。原因：单用户场景下轮询的资源开销可忽略，且实现复杂度远低于 SSE。  
- **每步骤更新 `current_step_label`** 的时机（与前端 G10 文案表对齐）：
  - `framework` job：「正在分析场景需求…」→「正在生成剧情框架…」→「正在生成参与角色…」→ 完成
  - `world` job：循环每节两步 ——「正在生成第 {ch} 章第 {sec} 节场景…」→「正在为第 {ch} 章第 {sec} 节生成任务目标…」→ 下一节 → 完成

## 8. 状态机与运行规则（实现要点）

1. **`lifecycle_phase`（6 值）**：严格实现 PRD §5.4 迁移；`creation_running` 与 Job 状态可并行，但产品展示以 Job 为准；**不**实现 `runtime_complete` 与 `archived`。  
2. **`runtime_awaiting_user`**：与 §6.6.4 及最后一条 `turn` 的 `expects_user_response` 一致；**所有**写 `turn` 的入口（自动开场、用户发送、NPC 自动续聊）须走同一领域服务以防分叉。  
3. **P3a 切节**：事务内更新 `current_chapter_id`/`current_section_id`；服务端**不**清空任何小节 `turns.jsonl`；切到 `turn === 0` 的小节时同步执行 §6.6.5 自动开场。  
4. **R1/R2**：异步任务不持有对话写锁；失败不回滚 `lifecycle_phase`。

### 8.1 进节自动开场（§6.6.5）实现细节

- **触发点**：`POST .../enter` 接口内部判断 `turns.jsonl` 行数为 0 时直接调用 `AutoOpenerService.generate_opener()`，**同步**返回；不另起异步 Job（用户已在等待界面）。  
- **提示词**：`backend/app/prompts/auto_opener.md`，输入包括 `section_narrative`、`section_mission`、`character_roster`、`enriched_scene_description`、`appearing_npc_ids`；输出严格按 §6.6.3 `turn` 结构。  
- **NPC 选择策略**：若 `appearing_npc_ids.length === 1`，使用该 NPC；若 `length === 2`，**默认使用 `appearing_npc_ids[0]`**（即稳定顺序首位，与 G8 默认接收人保持一致，体感连贯）。**`length === 0` 的情况首版不可能出现**——PRD §6.4.3 v0.5.1 起已硬性要求每节至少 1 名 NPC 出场，由剧情扩写智能体的提示词+校验层双重保证；本服务无需再处理该分支。  
- **写入语义**：`speaker_id = 选定 NPC`、`recipient_id = "user"`、`expects_user_response = true`、`turn_writer = "model_npc"`、`created_at = now`；写入 `turns.jsonl` 首行；`package.json.runtime_awaiting_user` 更新为 `true`。  
- **失败行为**：不写入半成品；返回 5xx + `error_code = "auto_opener_failed"`；前端展示重试按钮，再次调用 `POST .../auto-opener` 即可重试（**不**走 `enter`）。

### 8.2 用户发送消息后的 NPC 续聊

- NPC 续聊提示词模板 `backend/app/prompts/dialogue_npc.md`，输入含 `prior_turns`（截至此刻）、`section_narrative`、`section_mission`、`character_roster`。

### 10.2 单小节长历史的上下文裁剪策略

**问题**：用户在一节内可能连续聊很多轮，全部塞进 LLM 上下文会超长且变慢。

**首版策略**（简单、配置化）：

- 每次调用 §6.6 续聊 / §6.6.5 自动开场 / §6.7 提示 / §6.8 复盘时，从 `turns.jsonl` 取**最近 `DIALOGUE_HISTORY_LIMIT` 条**（默认 **20**，环境变量可调），按时间升序拼入 LLM 上下文。
- 上文锚点（`section_narrative` / `section_mission` / `character_roster` / `enriched_scene_description`）**始终全量**注入；它们提供"这一节我们在演什么"的稳定背景。
- **§6.8 本节复盘**例外：复盘需要更完整的回顾，可取**最近 `DIALOGUE_HISTORY_LIMIT * 2 = 40` 条**（或独立配置）。
- 用户**真的**在一节中聊了 100+ 条还不想结束时——根据 PRD（用户可无限对话），LLM 仍然只看最近一段，前面历史"逐渐淡出"是可接受的；前端聊天区仍**全量展示**所有历史给用户。
- 不实现摘要替代远古历史（复杂度高、易丢信息），由用户自己用"删除场景包"再开新包来重置。

### 11.2 双分支测试-合并工作流（强约定）

> **首次部署的特殊性**：仓库的 `git clone` 默认拉 main 分支。**第一个里程碑（M0）首次落到服务器时，main 还只有文档没有代码**——代码在 feature 分支上等用户验收。所以**首次部署须按部署指南步骤 3.5 显式 `git checkout <feature 分支>`** 一次，之后才能进入下方的稳态循环。M0 验收合并后此问题消失，但**每个新的 feature 分支首次拉到服务器测试时，仍走"`pull-and-restart.sh <分支名>`"切换分支**。

每一轮工程迭代严格按以下顺序：

```
我（Cloud Agent）                       你（腾讯云 Web Shell + 手机）
─────────────────────                  ──────────────────────────────
1. 在 feature 分支开发并推送 GitHub
   （分支名由 Agent 给出，形如 cursor/m1-scenario-crud-a4d3）
                                        2. 在 Web Shell 拉取该 feature 分支测试：
                                           cd /opt/gcp
                                           ./deploy/pull-and-restart.sh \
                                              cursor/<Agent-给你的分支名>
                                        3. 手机浏览器测试 http://43.155.205.89
                                        4. 通过 → 告诉我；不通过 → 我修
5. 我用 git merge --no-ff 合并到 main
   并删除该 feature 分支
                                        6. 在 Web Shell 拉取 main 作为基线：
                                           cd /opt/gcp
                                           ./deploy/pull-and-restart.sh
```

**`pull-and-restart.sh` 行为约定**（持续加固）：

- 接受**一个可选参数**：要拉取的远端分支名，**不传则默认 `main`**。
- 内部步骤：① 校验远端存在 `origin/<branch>` → ② `git fetch` + **`git checkout -B <branch> origin/<branch>`**（工作区与**本地分支名**同时与远端对齐；避免旧版仅 `reset --hard` 导致「磁盘已是新 commit、但 `git branch` 仍显示旧 feature 名」的困惑）→ ③ 按需 `pip install` → ④ 按需 `npm ci && npm run build` → ⑤ **`systemctl restart gcp-backend` 为硬门槛**（未注册单元则**脚本失败退出**，防止「只构建了前端、后端进程仍是旧代码」的假成功）→ ⑥ **本机 `curl /api/v1/health` 必须成功**（确认 uvicorn 新进程已响应）→ ⑦ `reload nginx` → ⑧ 打印分支名、commit、最近一条提交说明。
- **为何前端会变、接口不变**：Nginx 对 `/debug/`、`frontend/dist` 多为直接读磁盘；`/api/` 走反代到本机 Python 进程，**不重启 gcp-backend 则内存仍是旧路由**。运维指南「排错」表已单列此现象。
- 每次执行后**清晰打印**当前分支名、commit、最近提交，便于与 GitHub 对照。

**为什么这条流程是值得遵守的**：

- 单用户、单服务器，**main 永远是经过你验收过的"已知好状态"**；当某一轮 feature 出问题，你只要重跑 `./deploy/pull-and-restart.sh`（不带参数）就立刻回滚到上次的 main，零成本。
- 不引入 staging / production 区分、不引入 CI/CD、不引入 Webhook——所有触发都由你**主动一条命令**控制，最透明。
- 不是浪费时间——拉一个 feature 分支重启一次服务全过程不到 30 秒。

> **GitHub Webhook 自动 pull**：首版**不做**。原因同上：透明度优先。

### 11.6 超时链统一配置（应对 LLM 慢响应）

**问题**：单次 LLM 调用 5~60 秒；最坏情形（NPC 续聊 3 条 + 单条复杂）逼近 90 秒；Nginx 默认 `proxy_read_timeout=60s` 会**误杀**长任务返回 504。

**首版统一超时链**（**从内到外，逐层放宽**）：

| 位置 | 超时值 | 实现 |
|------|--------|------|
| LLM 客户端单次 HTTP 调用 | **120 秒** | `httpx.AsyncClient(timeout=httpx.Timeout(120.0))`，含连接 / 读 / 写 |
| LLM 客户端"业务超时"（单逻辑动作） | **150 秒** | 比 HTTP 略高，留余量给"修复重试"一轮 |
| Uvicorn `--timeout-keep-alive` | **180 秒** | 启动参数 |
| Nginx `proxy_read_timeout` / `proxy_send_timeout` | **180 秒** | §11.4 配置已含 |
| Nginx `keepalive_timeout` | **90 秒** | 短连接，浏览器自然重连即可 |
| 前端 `fetch` AbortController | **180 秒** | `apiClient.ts` 中统一封装 |
| 前端"长任务体感文案"切换 | 10/30/60 秒梯度 | 见下 |

**超时归属语义**（哪个层先到时间谁先报错）：

- 任何 LLM 单次 > 120s → `LlmClient` 抛 `LlmTimeoutError`，Service 决定是否做 1 次修复重试。
- LLM 总时长 > 150s（含重试）→ Service 抛 `BusinessTimeoutError`，Router 返 HTTP 504。
- Nginx 上游空闲 > 180s → Nginx 返 HTTP 504 给前端（理论上 Service 应该在此之前已经收到 timeout 并主动结束）。
- 前端 fetch > 180s → AbortController 主动取消，业务 Hook 抛"操作超时，请重试"。

**用户体感文案（前端 typing 指示器旁的副文案）**：

| 等待时间 | 副文案 |
|---|---|
| 0~10s | （无副文案，typing 三个点足够） |
| 10~30s | "正在思考……" |
| 30~60s | "正在生成回应……" |
| 60~120s | "复杂任务，请稍候……" |
| > 120s | （仍展示等待，**不**升级文案；超过 180s 进失败态） |
| 失败态 | "NPC 没及时回应，点这里重试" + 按钮 |

**何时不该用同步等待**：

- §6.4 + §6.5 全书循环（每节 30~90s × 最多 20 节）= 累计**几分钟到十几分钟**，**必须**走异步 Job + 轮询，不能在 HTTP 上等。
- §6.8 复盘**累计回合很多**时单次可能 60~120s——**首版**仍用同步等待（用户主动点击触发，能接受），如果实测>120s 频繁出现，再改异步 Job。
- §6.2 + §6.3（框架 + 角色）= 30~90s，由于已经在 Job 里所以同样走异步。

**长任务 Job 的轮询协议**：

- 前端 `useFrameworkJob` / `useWorldJob` 每 **1 秒** 轮询 `GET .../jobs/{id}`。
- 服务端 `Job.status` ∈ `queued | running | succeeded | failed | canceled`，附 `current_step_label` 文案给 G10。
- 单个 Job 最大执行时长**软上限 30 分钟**——超过则标记为 `failed`，附 `error_code = "job_exceeded_max_duration"`。
- 用户在 P2.2 / P2.4 点"取消" → 前端调 `POST .../jobs/{id}/cancel` → 服务端置 `canceled` 并清掉已落的部分中间态（按 G3/G4 规则）。

> 这套超时链是"诚实工程"：**不假装快**，**也不让用户卡死**；当真的失败时**清楚告诉用户失败原因 + 给重试按钮**。

## 12. 风险与待决事项

| 项 | 风险 / 说明 | 建议 |
|----|-------------|------|
| **LLM 超时（已在 §11.6 系统化解决）** | 单次 5~60s、最坏 90s；Nginx 默认 60s 会误杀 | 采用 §11.6 超时链：LLM 120s / Uvicorn 180s / Nginx 180s / 前端 180s；前端梯度副文案 |
| **DeepSeek 输出不符合 schema** | LLM 偶尔输出非 JSON 或字段错位 | 每个智能体输出端加 `Validators`，失败则触发 1 次"修复重试"（把错误信息塞进 prompt 再次请求）；仍失败返回 5xx |
| **进节自动开场被反复触发** | 用户网络抖动反复打开 P3 | `POST .../enter` 在该节 `turns.jsonl` 已存在 ≥1 行时**不**触发自动开场（幂等） |
| **章节自由切节导致 NPC 上下文跳脱** | 用户跳节后 NPC 不知"上一节发生了什么" | PRD §5.7 分区原则：每节的 NPC 只读本节 `prior_turns`，不读其他节；这是产品特性而非 bug |
| **独白节（已由 PRD 禁止，不再是风险）** | PRD v0.5.1 §6.4.3 硬性禁止 `appearing_npc_ids.length === 0` | 提示词显式约束 + Validators 兜底；若 LLM 偶发输出空，触发 1 次修复重试，仍空则视为生成失败回到 `creation_failed` |
| **文件并发写入** | 同一包并发请求可能撕裂 JSON | 进程内 `asyncio.Lock` 按 `scenario_id` 加锁；首版单用户低风险 |
| **磁盘填满** | 每节对话无限累积 | 单用户 Demo 风险低；监控 `du -sh data/` 即可 |
| **DeepSeek API Key 泄漏** | 误提交到 GitHub | `.env` / `/etc/gcp/backend.env` 不入仓；`.gitignore` 显式列出 |
| **HTTP 明文风险（首版无 HTTPS）** | 用户在公网下用 IP+HTTP 访问，对话内容明文传输 | 单用户 Demo 阶段可接受；用户购置域名后立刻按 §11 添加 Let's Encrypt 证书启用 HTTPS |
