## 2. `lifecycle_phase` 状态机

### 2.1 6 个状态的语义回顾（PRD §5.4）

```
draft → intake_committed → creation_running → creation_succeeded → runtime_active
                                ↓                                         ↑
                          creation_failed ────────────────────────────────┘
                                ↑                                         │
                                └─── 用户修改 intake 重提交 ──────────────┘
```

### 2.2 状态迁移表

| 当前状态 | 触发事件 | 下一状态 | 副作用 |
|---|---|---|---|
| 〈不存在〉 | `POST /scenario-packages` | `draft` | 创建 `data/scenarios/{id}/package.json`，更新 `index.json` |
| `draft` | `POST .../commit-intake` 成功（含扩写） | `intake_committed` | 落盘 `intake.json` + `analysis.json` |
| `draft` | `DELETE` | 〈不存在〉 | 删除整个包目录 |
| `intake_committed` | `POST .../jobs/framework` | `creation_running` | 创建 framework job |
| `intake_committed` | `POST .../commit-intake`（重新提交，无重置需求） | `intake_committed`（自迁，仅刷新 `intake.json` + `analysis.json`） | 不清下游产物 |
| `creation_running` | framework job 失败 | `creation_failed` | job 转 `failed`；`current_step_label` 留作错误说明 |
| `creation_running` | framework job 成功 | `intake_committed`（**Note**：framework 成功**不**直接进 succeeded；framework + roster 落盘后回 `intake_committed`，等待 world job） | — |
| `intake_committed`（已有 framework+roster） | `POST .../jobs/world` | `creation_running` | 创建 world job |
| `creation_running`（world job） | 全节 narrative+mission 落盘 | `creation_succeeded` | — |
| `creation_running`（world job） | 任一节失败 | `creation_failed` | — |
| `creation_running`（任一 job） | `POST .../jobs/{id}/cancel` | `intake_committed` | 按 §6 / §7 清库范围 |
| `creation_failed` | `POST .../commit-intake`（含 `force_reset_creation=true`） | `intake_committed` | G3 清库（§6） + 重新扩写 |
| `creation_failed` | `POST .../jobs/framework` 或 `.../jobs/world` 重试 | `creation_running` | — |
| `creation_succeeded` | `POST .../sections/{ch}/{sec}/enter` 首次 | `runtime_active` | 更新指针、若目标节无 turn 触发自动开场 |
| `creation_succeeded` | `POST .../commit-intake`（`force_reset_creation=true`） | `intake_committed` | G3 清库（含已 succeeded 的全部产物） |
| `runtime_active` | `POST .../sections/.../enter` 任意切节 | `runtime_active`（自迁） | 仅更新指针 + 必要时自动开场 |
| `runtime_active` | `POST .../turns` | `runtime_active`（自迁） | 追加 turn |
| `runtime_active` | `DELETE` | 〈不存在〉 | 删除整个包 |

> **重要**：`runtime_complete` 与 `archived` 在 v0.5.0 已彻底移除（PRD §5.4）。`runtime_active` 是终止状态（直到用户主动 DELETE）。

### 2.3 状态机校验函数（Validators 层）

实现位置：`backend/app/validators/lifecycle.py`

```python
def assert_can_transition(current: LifecyclePhase, action: LifecycleAction) -> None:
    """
    若不允许迁移则抛 LifecyclePhaseError(error_code='lifecycle_phase_invalid', details={...})
    """
```

所有 Service 在执行业务动作之前**必须**先调本函数。Router 层将抛出的 `LifecyclePhaseError` 翻译成 409 响应。

## 3. Job 状态机

### 3.1 状态枚举

```
queued → running → succeeded
              ↓
              ├→ failed
              └→ canceled
```

| 状态 | 含义 |
|---|---|
| `queued` | 已落盘 `jobs/{id}.json`，尚未开始执行 |
| `running` | 后台任务运行中 |
| `succeeded` | 全部产物落盘成功 |
| `failed` | LLM 或落盘失败；`error_code` / `error_message` 填充 |
| `canceled` | 用户取消；按 §6/§7 清库 |

`succeeded` / `failed` / `canceled` 为 **terminal** 状态，不可再变。

### 3.2 Job 类型

| `type` | 服务方法 | 步骤 |
|---|---|---|
| `framework` | `JobRunner.run_framework()` | ① §6.2 framework → ② §6.3 roster |
| `world` | `JobRunner.run_world()` | ① 按 framework 中节顺序循环：A. §6.4 narrative；B. §6.5 mission |

### 3.3 取消语义

- 取消请求到达时，若 Job 正在某次 LLM 调用中——**不**中断该次调用（DeepSeek HTTP 已发，等返回再 discard）；
- 若该次 LLM 返回成功——产物**不**落盘；
- Job 状态转 `canceled`，按 §6/§7 清库。

### 3.4 进度文案 `current_step_label`

| Job 类型 | 阶段 | 推荐文案 |
|---|---|---|
| `framework` | step 1 开始 | `"正在生成剧情框架…"` |
| `framework` | step 2 开始 | `"正在生成参与角色…"` |
| `world` | 进入第 ch 章第 sec 节 step A | `"正在生成第 {ch} 章第 {sec} 节场景…"` |
| `world` | step B | `"正在为第 {ch} 章第 {sec} 节生成任务目标…"` |
| 任意 Job | 完成 | `"完成"`（前端可不再展示） |
| 任意 Job | 失败 | `"生成失败：{error_message}"` |

每次更新 `current_step_label` 时同步更新 `updated_at` 与 `progress_hint`（如 "3/6"）。

## 4. §6.6.4 回合规则的执行算法

PRD §6.6.4 列出 9 条规则（编号 0~8）。在 `DialogueService` 写入任何一条新 turn 之前，**按顺序**做以下校验。任一失败则抛 `InvalidTurnError`，Router 翻译为 422。

### 4.1 写入用户 turn 时的校验序

输入：`scenario_id, chapter_id, section_id, content, recipient_id`

```text
1. 加载 package.json 与 section_narrative
2. 校验规则 0：(chapter_id, section_id) === (current_chapter_id, current_section_id)
3. 校验 runtime_awaiting_user === true（隐含规则 1 的前置）
4. 校验最近一条 turn.expects_user_response === true（隐含规则 1）
5. 校验 recipient_id ∈ section_narrative.appearing_npc_ids ∪ {"user"} 集合
   - 实际：recipient_id 必须 ∈ appearing_npc_ids（不允许给自己发）
6. 校验 content.length ∈ [1, 8000]
7. 构造新 turn：
   - speaker_id = "user"
   - turn_writer = "human_user"  (规则 6.6.1 交叉)
   - expects_user_response = false（用户发完话，期望 NPC 回应）
   - created_at = now()
   - turn_id = uuid4()
8. 追加写入 turns.jsonl
9. 更新 package.json：runtime_awaiting_user = false（用户已发，轮到 NPC）
10. 异步触发 hint stale 迁移（详见 §10）
```

### 4.2 写入 NPC turn 时的校验序

NPC turn 可能由 3 个来源触发：

- A. §6.6.5 自动开场（**首条** NPC turn）
- B. §6.6 用户发送后的续聊（1~3 条）
- C. 后续如果引入"NPC 主动追问"——首版无此场景

对每一条候选 NPC turn：

```text
1. 加载 turns.jsonl 末尾的最近 K 条作为 prior_turns 上下文
2. 校验规则 0：chapter_id / section_id === current pointer
3. 计算 L = 末尾连续 NPC–NPC turn 数（speaker !== user 且 recipient !== user）
4. 若候选 turn 也是 NPC–NPC（speaker !== user 且 recipient !== user）：
     校验规则 7：L + 1 ≤ 3
     若违反 → 终止本条候选写入，强制让下一条 NPC turn 的 recipient = "user"
5. 校验 speaker_id ∈ section_narrative.appearing_npc_ids
6. 校验 recipient_id ∈ section_narrative.appearing_npc_ids ∪ {"user"}
7. 校验 speaker_id !== recipient_id（规则 5.2 NPC id 唯一性的延伸）
8. 校验 content.length ∈ [1, 8000]
9. 校验 expects_user_response 与 recipient_id 的一致性（规则 2、4）：
   - expects_user_response === true ⟹ recipient_id === "user"
10. 构造 turn 对象 (turn_writer = "model_npc", created_at = now, turn_id = uuid4())
11. 追加写入 turns.jsonl
12. 更新 package.json.runtime_awaiting_user：
    - 若候选 turn.expects_user_response === true → true (规则 5)
    - 否则 → false (规则 4)
```

### 4.3 §6.6.4 规则 8 的实现（交回练习者主动权）

实现在 `DialogueService.decide_next_npc_action()` 中——决定"下一条 NPC turn 是否继续 NPC–NPC、还是必须转向 user"：

```text
def decide_next_npc_action(prior_turns, appearing_npc_ids) -> Decision:
    L = trailing_npc_npc_count(prior_turns)
    if len(appearing_npc_ids) < 2:
        # 单 NPC 节，所有 NPC turn 都是 NPC→user
        return Decision(force_to_user=True)
    if L >= 2:
        # 已经连续 2 条 NPC–NPC，强制下一条转向 user (规则 8)
        return Decision(force_to_user=True)
    if L >= 3:
        # 不应到达此分支（规则 7 已阻拦）；保险起见
        raise InvariantViolation("L >= 3 should never happen")
    # L ∈ {0, 1}，允许 NPC–NPC 或 NPC→user，由生成提示词自由发挥
    return Decision(force_to_user=False)
```

## 5. §6.6.5 进节自动开场流程

### 5.1 触发条件

由 **`POST .../sections/{ch}/{sec}/enter`** 接口与 **`POST .../sections/{ch}/{sec}/auto-opener`** 接口共同触发。逻辑封装在 `AutoOpenerService.generate_opener()`。

### 5.2 触发流程

```text
def enter_section(scenario_id, ch, sec):
    pkg = package_repo.load(scenario_id)
    assert_can_transition(pkg.lifecycle_phase, EnterRuntime)
    assert framework.has_section(ch, sec)  # 否则 404 section_not_found

    # 1. 更新指针
    pkg.current_chapter_id = ch
    pkg.current_section_id = sec
    if pkg.lifecycle_phase == "creation_succeeded":
        pkg.lifecycle_phase = "runtime_active"
    pkg.updated_at = now()
    package_repo.save(pkg)

    # 2. 检查目标节是否需要自动开场
    turns = turn_repo.list_section(scenario_id, ch, sec)
    if len(turns) > 0:
        return RuntimeResponse(turns=turns, auto_opener_triggered=False)

    # 3. 调自动开场（同步阻塞）
    try:
        new_turn = auto_opener_service.generate_opener(scenario_id, ch, sec)
    except LlmTimeoutError:
        raise  # 504
    except (LlmFailure, SchemaValidationError):
        raise AutoOpenerFailedError()  # 500

    # 4. 写入首条 turn（走完整 §6.6.4 校验）
    dialogue_service.write_turn(new_turn)

    # 5. 更新 awaiting flag
    pkg.runtime_awaiting_user = True
    package_repo.save(pkg)

    return RuntimeResponse(
        turns=[new_turn],
        auto_opener_triggered=True,
        auto_opener_turn_id=new_turn.turn_id,
    )
```

### 5.3 失败处理

- LLM 超时 → 抛 504，前端按 §11.6 文案展示"NPC 没及时回应"+ "重试"按钮 → 用户点击 → 前端调 `POST .../sections/{ch}/{sec}/auto-opener` 重试。
- LLM 返回但 schema 校验失败 → AutoOpenerService 内部做**最多 1 次修复重试**（把 schema 错误塞进 prompt 再请求一次）→ 仍失败抛 500。
- 指针**已经更新**至目标节（用户切节意图已生效），但 `turns` 为空——这是合法的中间态。

### 5.4 NPC 选择策略

- `len(appearing_npc_ids) === 1` → 使用该 NPC
- `len(appearing_npc_ids) === 2` → 使用 `appearing_npc_ids[0]`（保持稳定首项，与 G8 默认接收人一致）
- `len(appearing_npc_ids) === 0` → **不可能**（PRD v0.5.1 §6.4.3 已硬性禁止）；防御性代码遇到时抛 500 `auto_opener_failed`

## 6. G3「重置创作」清库范围

### 6.1 触发场景

`POST .../commit-intake` 请求体 `force_reset_creation === true` 时执行。

### 6.2 清除清单（按顺序，原子）

> **原子**：若中途任一步失败，须 rollback 已删除的部分（首版用"先 rename 到 .trash 再统一删除"模拟原子性）。

```
1. data/scenarios/{id}/framework.json
2. data/scenarios/{id}/roster.json
3. data/scenarios/{id}/sections/ 整个目录（含 narrative.json / mission.json / turns.jsonl / hints/ / analytics.json）
4. data/scenarios/{id}/jobs/ 整个目录
```

```
- data/scenarios/{id}/package.json（更新 lifecycle_phase 为 intake_committed、清空 current_chapter/section）
- data/scenarios/{id}/intake.json（会被新的 intake 覆盖写）
- data/scenarios/{id}/analysis.json（会被新的扩写结果覆盖写）
- data/scenarios/index.json
```

### 6.4 lifecycle_phase 副作用

无论调用前 phase 是什么，G3 完成后 phase **都**回到 `intake_committed`。

## 8. 用户发消息 → NPC 续聊算法

### 8.1 流程

```text
def send_user_message(scenario_id, ch, sec, content, recipient_id):
    pkg = package_repo.load(scenario_id)
    assert_can_transition(pkg.lifecycle_phase, SendUserMessage)
    assert (pkg.current_chapter_id, pkg.current_section_id) == (ch, sec)
    assert pkg.runtime_awaiting_user

    # 1. 写入用户 turn（§4.1 校验序）
    user_turn = build_user_turn(scenario_id, ch, sec, content, recipient_id)
    dialogue_service.write_turn(user_turn)

    # 2. 异步触发 hint stale 迁移
    hint_service.mark_stale_async(scenario_id, ch, sec, before_turn_id=user_turn.turn_id)

    # 3. 同步生成 NPC 续聊 1~3 条
    new_turns = [user_turn]
    while True:
        decision = dialogue_service.decide_next_npc_action(
            prior_turns=load_prior_turns(scenario_id, ch, sec, limit=DIALOGUE_HISTORY_LIMIT),
            appearing_npc_ids=section_narrative.appearing_npc_ids,
        )
        npc_turn = dialogue_service.generate_npc_turn(
            scenario_id, ch, sec, force_to_user=decision.force_to_user,
        )
        dialogue_service.write_turn(npc_turn)
        new_turns.append(npc_turn)

        # 停止条件:
        # (a) 该条 NPC turn.expects_user_response === true → 轮到用户
        if npc_turn.expects_user_response:
            break
        # (b) 总 NPC 续聊条数（不含用户那条）已达 3 条上限
        if len(new_turns) - 1 >= 3:
            break
        # (c) 单 NPC 节情况下，NPC 一条说完就应该等用户（生成提示词应自然达到此）
        if len(appearing_npc_ids) == 1:
            # 防御性：如果模型输出 expects_user_response=false，强制改为 true
            if not npc_turn.expects_user_response:
                # 不重写已落盘的 turn，而是直接跳出
                break

    pkg.runtime_awaiting_user = last_turn(new_turns).expects_user_response
    package_repo.save(pkg)
    return new_turns
```

### 8.2 停止条件总结

NPC 续聊**何时停**：

1. 任一 NPC turn `expects_user_response === true`（轮到用户）；或
2. 累计 NPC 续聊条数达 **3** 条上限（§6.6.4 规则 7）；或
3. LLM 失败 / 超时（抛错给用户，已落盘的 turn 保留）。

### 8.3 LLM 失败兜底

若在生成第 N 条 NPC turn 时 LLM 失败：

- 已写入的 1~(N-1) 条**保留**（包括用户那条）；
- 抛 500 `llm_failure`，响应中 `new_turns` 包含已写入的部分；
- 前端展示"NPC 没回应完，点击重新生成"——首版**不**实现该重试按钮（用户可手动再发一条消息触发新的续聊）。

## 9. LLM 输出 schema 校验与修复重试

### 9.1 问题

DeepSeek 偶尔输出非 JSON、字段错位、数组长度违规等。

### 9.2 处理流程（每个智能体调用通用）

```text
def call_llm_with_schema(prompt, response_model):
    for attempt in [1, 2]:  # 至多 2 次（首次 + 修复重试 1 次）
        raw = llm_client.chat(prompt)
        try:
            parsed = parse_json(raw)
            validated = response_model.model_validate(parsed)
            return validated
        except (JSONDecodeError, ValidationError) as e:
            if attempt == 1:
                # 修复重试：把错误信息塞进 prompt 后重试
                prompt = augment_prompt_with_error(prompt, raw, e)
                continue
            else:
                raise LlmSchemaFailure(detail=str(e))
```

实现位置：`backend/app/clients/llm_client.py`。

### 9.3 适用范围

所有调 LLM 输出结构化数据的智能体均走此流程：

- §6.1 扩写（输出 `scenario_analysis`）
- §6.2 framework
- §6.3 roster
- §6.4 narrative
- §6.5 mission
- §6.6 NPC turn（输出单条 turn JSON）
- §6.6.5 自动开场（输出单条 turn JSON）
- §6.7 hint
- §6.8 analytics

## 10. R1 提示对象的 stale 迁移

### 10.1 触发时机

每次用户成功写入一条 `speaker_id === "user"` 的 turn 后，**异步**触发：

```text
所有 linked_turn_id ∈ {turn_id | turn 是用户 user_turn 之前的 NPC turn 且 expects_user_response === true}
的 hint 对象 → hint_status 置为 "stale"
```

### 10.2 实现策略

- `HintRepo.list_section_hints(ch, sec) → list[Hint]`
- 对每条 hint，若其 `linked_turn_id` 早于本次 user_turn 的 `created_at` → 标记 stale 并覆盖写回 JSON 文件。
- 实际首版**简化**：直接把该节 `hints/` 目录下所有 `hint_status === "ready"` 的 hint 改成 `"stale"`。原因：单用户场景下不会高频生成 hint，逐一比较时间戳意义不大。

### 10.3 GET hints/latest 行为

- 返回该节**最新一条**（按 `generated_at` 倒序）hint；包含 `hint_status` 当前值（可能是 `ready` 或 `stale`）。
- 前端按 `hint_status` 决定展示样式（PRD §6.7.1）。

## 11. 进节 / 切节流程总结

### 11.1 三种进入入口

| 入口 | 触发接口 | 默认指针目标 |
|---|---|---|
| P2.5「直接探索世界」 | `POST .../sections/1/1/enter` | `S[0]`（第 1 章第 1 节） |
| P2「进入场景」 | 前端先 `GET .../runtime` → 拿到 `current_chapter/section` → `POST .../sections/{cur}/enter` | 包当前指针；若指针为 null 则 `S[0]` |
| P3a「确认进入」 | `POST .../sections/{选中节}/enter` | 用户在 P3a 选中的节 |

### 11.2 切节副作用一览

| 副作用 | 触发条件 |
|---|---|
| 更新 `current_chapter_id` / `current_section_id` | 总是 |
| `lifecycle_phase: creation_succeeded → runtime_active` | 仅首次进入 |
| 自动开场（写入首条 NPC turn） | 目标节 `turns.length === 0` |
| 清空前端 P3 输入框草稿 | 由前端处理（不调后端） |
| 清空任何小节历史 | **绝不**（即使切到 `t < i` 也保留历史） |

### 11.3 用户在节内连续多次"切回当前节"

- `POST .../sections/{ch}/{sec}/enter` 总是幂等：
  - 若指针已经是 `(ch, sec)` 且 turn 数 ≥ 1 → 只返回当前节快照，不触发开场，不修改任何状态；
  - 若指针已经是 `(ch, sec)` 但 turn 数 === 0 → 触发自动开场（与第一次进入一致）；
  - 若指针不是 `(ch, sec)` → 更新指针，按 §11.2 表执行。

## 12. 错误兜底总结

| 错误层级 | 处理位置 | 用户感知 |
|---|---|---|
| HTTP 协议错误（400 invalid_request_body） | Router 入参校验 | Toast: "请求格式错误" |
| 业务前置不满足（409） | Validators / Service | Toast: 中文错误描述 |
| 字段约束（422） | Pydantic + Validators | 表单字段下提示 / Toast |
| LLM 超时（504） | LlmClient → Service → Router | Typing 指示 + "NPC 没回应，重试"按钮 |
| LLM schema 失败（500 llm_failure） | LlmClient | 同上 |
| 自动开场失败（500 auto_opener_failed） | AutoOpenerService | "重试本节开场"按钮 |
| 文件 I/O 失败（500 repository_io_error） | Repository | Toast: "系统异常，请稍后重试" |
| 未分类（500 internal_error） | 全局 Exception Handler | Toast: "系统异常，请稍后重试" |

## 13. 验收清单（开发者自检）

在 M5 阶段完成"后端跑通全链路"时，应能在 `/debug/` 调试页**完整**走完以下序列且每一步符合本文描述：

- [ ] 新建空包 → `package.json` 出现，`lifecycle_phase === "draft"`
- [ ] 提交五字段 → `intake.json` + `analysis.json` 出现，phase = `intake_committed`
- [ ] 启动 framework job → 轮询过程中 `current_step_label` 切换两次 → 成功后 `framework.json` + `roster.json` 出现
- [ ] 启动 world job → 轮询过程中 `progress_hint` 从 `1/N` 递增到 `N/N` → 成功后每节目录均有 `narrative.json` + `mission.json`，phase = `creation_succeeded`
- [ ] 调 enter `(1, 1)` → 触发自动开场 → `turns.jsonl` 出现首条 NPC turn，phase = `runtime_active`，`runtime_awaiting_user = true`
- [ ] 发送用户消息 → `turns.jsonl` 追加用户 turn 与 1~3 条 NPC turn → 最后一条 `expects_user_response` 必为 true（除非用户已发送终结性内容，但 NPC 应继续邀请）
- [ ] 调 hints → `hints/{turn_id}.json` 出现 ready 状态
- [ ] 再发一条用户消息 → 旧 hint 自动转 stale
- [ ] 调 analytics → `analytics.json` 出现 ready 状态
- [ ] 切节到 `(1, 2)` → 指针更新、触发自动开场、`(1, 1)` 历史保留
- [ ] 切回 `(1, 1)` → 指针更新、**不**重复自动开场、历史完整还原
