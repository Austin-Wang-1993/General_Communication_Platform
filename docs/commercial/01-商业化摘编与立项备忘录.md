## 2. 产品价值主张（摘编）

- **一句话**：用户用自然语言描述场景与目标 → 系统生成带章节与 NPC 的英文练习世界 → **进节即由 NPC 开场** → 用户用英文对话推进 → **双轨辅助**（回答提示 R1、本节复盘 R2）与可选**句级翻译**降低认知门槛。  
- **三个核心抽象**（与 README、中台一致）：**场景包隔离**；**章节自由推进**（非线性通关）；**进节自动开场**（避免空屏）。  
- **差异化记忆点**：同一产品内把「剧情沉浸」「英语口语练习」「可审计的机读状态机」绑在一条链路上；多方 NPC 时仍保持**单条台词单接收方**等可验收规则（见中台 §6.6.6 与前端 F-P3-10）。

## 3. 当前 Demo 的「已实现」与「刻意未做」

### 3.1 已实现（可作为验证与演示的资产）

- 场景包全链路：录入 → 框架/世界异步 Job → 运行期 **runtime / enter / turns** → **R1 hints / R2 analytics**。  
- C 端主路径页面（欢迎、列表、向导、预览、P3 对话、P3a 选节等，详见前端需求文档与 README）。  
- **文件型持久化**：`data/scenarios/{scenario_id}/...` 便于 PM 与合伙人**直读 JSON** 审阅生成质量。  
- **一键部署脚本**：降低「非技术合伙人」侧的环境同步成本（见运维指南与 `deploy/server-one-shot-sync.sh`）。

### 3.2 刻意未做或非目标（对产品文档中「非目标 / MVP 裁剪」的归总）

下列项在现有产品或技术文档中多已标注为**非目标**、**首版不做**或**未来范围**，商业化讨论时需**显式承认**，避免对外承诺过度。

| 能力域 | 当前状态（摘编） | 商业化含义 |
|--------|------------------|------------|
| **多租户、账号、权限** | API 文档：首版无鉴权；技术方案：单用户假设 | 无法直接做按用户计费、企业 SSO；需另立项目 |
| **支付与订阅** | 未在需求中定义 | 收入模型需从零设计系统承载 |
| **数据库** | 首版纯 JSON 文件 | 并发、查询、报表需迁移或叠加存储 |
| **HTTPS / 域名** | 首版纯 IP + HTTP（技术方案 §11） | 正式对客需证书与合规评估 |
| **语音、离线客户端** | 非目标（技术方案 §2） | 赛道扩展需单独预算与团队 |
| **旅程级报告、归档态** | 中台已取消 `runtime_complete` / `archived` 等 | 产品叙事上强调「练习可反复、无强制通关」 |
| **清空本节对话等** | 中台曾允许但首版未实现 | 运营/客服型能力待排期 |
| **翻译结果持久化** | §6.9：Demo 不写入 turns；弹窗合订为展示辅助 | 可产品化为「学习档案」能力 |

## 5. 商业化议题清单（讨论用，非决策）

以下条目**不**代表已选定的商业模式，仅用于立项会与合伙人**对齐议题**。

1. **目标客户**：B2C 自学者、B2B 企业培训、教育机构白标——与当前「单用户 Demo」差距多大？  
2. **计费单元**：按场景包数、按 LLM token、按 seat、按机构包年——与现有「无账号」架构的映射关系。  
3. **数据与隐私**：用户描述的场景文本、对话记录、复盘内容的归属与删除策略（GDPR/个保法语境下的最小必要原则）。  
4. **内容安全与合规**：用户生成 UGC 场景、NPC 台词的审核策略与责任边界。  
5. **从 Demo 到 MVP 的工程里程碑**：账号体系、HTTPS、数据库、监控与计费中台的**先后次序**（需另文或另表）。  
6. **护城河**：提示词与校验规则资产、行业模板库、教研标注数据、与企业 LMS 集成深度等。

## 6. 「为什么没做 / 若要做」清单（简表）

| 主题 | 为什么当前 Demo 没做（工程/产品原因摘要） | 若商业化要做（方向性，非工时承诺） |
|------|-------------------------------------------|--------------------------------------|
| 用户账号 | 降低首版验证复杂度，单用户够用 | 引入 IdP / 自建用户表、会话与鉴权中间件 |
| 数据库 | 快速迭代、可肉眼审 JSON | 选 OLTP 库 + 迁移脚本；或对象存储 + 索引服务 |
| HTTPS | 纯 IP 限制与首版演示优先 | 域名 + 证书 + Nginx 配置；HSTS 等 |
| 多租户隔离 | 场景包级隔离已实现；租户级未做 | 命名空间、配额、计费与数据隔离策略 |
| 计费用量 | 无服务端计量 | 网关日志、token 计数、套餐与欠费策略 |
| 高可用 | 单进程单服 | 负载均衡、无状态化、会话外置、队列化 Job |
| 翻译持久化 | 明确为 Demo 辅助 | 用户偏好表、学习档案、双语对照导出 |

## 7. 运维与交付摘编（对商务沟通的要点）

- 当前部署路径以 **腾讯云 Ubuntu + Nginx + systemd** 为主叙事；**一键同步脚本**降低「跟分支测试」的操作风险。  
- 正式商业化需补充：**监控与告警**、**备份与 RTO/RPO**、**密钥轮换**、**发布回滚**等（现有运维文档以「能跑通」为主，不覆盖完整 SRE 体系）。

## 8. 建议的对外介绍顺序（合伙人 / 技术朋友）

1. README **产品一句话 + 三个核心抽象**（30 秒）。  
2. 本文 **§3 边界 + §6 若要做**（对齐预期，避免「以为已能上生产」）。  
3. 中台文档 **§4 主线流程 + §5 生命周期**（证明产品逻辑完整）。  
4. 技术方案 **§2 目标与范围 + §11 部署**（证明工程可落地、已知局限）。
